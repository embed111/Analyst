# 需求详情-创建角色工作流与任务绑定-预稿

- 状态：预稿（待用户确认）
- 约束：本文件当前不合入现有正式需求索引，不替换既有正式需求文档。
- 对应原型预稿：`docs/workflow/prototypes/创建角色工作流-预稿/创建角色工作流预稿.html`

## 1. 主题目标与价值说明
1. 主题目标：
   1. 在 `训练中心` 内新增 `创建角色` 子模块，用对话式引导取代单次表单式录入。
   2. 允许用户通过文本描述与图片示例共同表达“想要什么角色”。
   3. 在用户完成描述后，将角色草案结构化为 `role_spec`，并基于该草案自动生成“创建角色工作流”关联任务。
   4. `创建角色` 页面只展示与当前角色创建工作流相关的任务引用，不展示任务中心的全部任务。
   5. 为角色新增与版本生命周期并行的运行状态轴 `runtime_status`，明确创建期、训练期、执行期与空闲期的任务门禁。
2. 业务价值：
   1. 把“角色需求表达 -> 创建任务生成 -> 创建过程追踪 -> 创建完成回空闲”收口为一条连续业务流。
   2. 避免 `训练中心`、`创建角色`、`任务中心` 分别维护多套任务真相源。
   3. 让角色创建过程可追踪、可留痕、可恢复，而不是只靠一次性表单或口头描述。
   4. 为后续把 `训练功能` 改造成同类“业务页引用任务底座”的模式建立模板。

## 2. 用户画像与使用场景
1. 用户画像 A：角色需求提出者
   1. 关注点：能像对话一样逐步说清楚角色目标、能力、边界与风格，不想一次性填满复杂表单。
2. 用户画像 B：workflow 操作者
   1. 关注点：创建角色后能直接看到当前工作流推进到了哪一步、卡在哪个任务，不需要切到全局任务中心才能理解进度。
3. 用户画像 C：平台治理者
   1. 关注点：角色的“版本状态”和“运行占用状态”必须分离，避免状态机混乱。
4. 核心场景：
   1. 用户进入 `训练中心 -> 创建角色`，通过对话输入角色目标、能力、边界、协作方式和示例图片。
   2. 系统将这些输入收口为结构化 `role_spec`。
   3. 用户启动角色创建工作流后，系统在任务中心生成一组关联任务。
   4. `创建角色` 页仅展示当前角色的创建任务流图、阶段、当前卡点和必要动作。
   5. 所有创建任务完成后，角色自动从 `创建中` 回到 `空闲`。

## 3. 用户旅程或关键流程
1. 用户进入 `训练中心`，切换到 `创建角色` 子模块。
2. 页面默认展示：
   1. 左侧：角色创建草稿/进行中列表；
   2. 中部：对话历史区 + 固定对话输入条 + 创建工作流阶段流 + 当前角色的关联任务流图；
   3. 右侧：角色草案摘要 + 选中任务详情 + 运行门禁说明。
3. 用户通过对话逐步提供：
   1. 角色名称；
   2. 角色目标；
   3. 核心能力；
   4. 禁止边界；
   5. 适用场景；
   6. 协作方式/输出风格；
   7. 可选图片示例。
4. 系统识别描述内容并实时生成结构化角色草案 `role_spec`。
5. 当系统判断信息已足够时，页面进入 `启动创建` 阶段，展示：
   1. 当前结构化角色草案；
   2. 预计生成的创建任务摘要；
   3. 当前仍缺失或不清晰的信息（若有）。
6. 用户点击 `开始创建` 后：
   1. 系统创建临时角色记录；
   2. 角色 `runtime_status` 变为 `创建中`；
   3. 系统在任务中心生成当前角色的创建任务图与依赖关系；
   4. `创建角色` 页开始按“仅当前角色”范围引用并展示关联任务。
7. 创建过程中：
   1. `创建角色` 页只展示当前角色创建工作流关联任务；
   2. 用户可查看当前阶段、任务详情、失败原因、执行证据摘要；
   3. 若需要查看全局任务图，可跳转到 `任务中心`，并自动带上当前角色/当前任务图过滤条件。
8. 当关联任务全部完成：
   1. 当前角色 `runtime_status` 自动切为 `空闲`；
   2. 页面显示“创建完成”；
   3. 角色进入后续可用状态，但其 `lifecycle_status` 仍按既有发布治理规则独立维护。

## 4. 功能需求清单（编号）
1. FR-RC-01 `训练中心` 新增 `创建角色` 子模块
   1. `训练中心` 顶部子模块至少调整为：
      1. `角色资产与版本`
      2. `创建角色`
      3. `训练闭环`
   2. `创建角色` 当前作为 `训练中心` 内业务子模块存在，不新增左侧独立顶层模块。
2. FR-RC-02 对话式角色需求收集
   1. `创建角色` 页必须以对话式收集为主入口，而不是一次性静态表单。
   2. 页面至少支持两类输入：
      1. 文本描述；
      2. 图片示例。
   3. 页面必须按轮次保留用户已输入的关键信息与系统追问结果。
   4. 页面必须显式展示真实可操作的对话输入入口，而不是只展示历史消息示意。
   5. 当角色已进入 `创建中` 后，固定输入条仍需保留，允许继续补充文本与图片示例，而不是在启动后直接消失。
   6. 当前预稿默认输入入口至少包含：
      1. 文本输入框；
      2. 图片上传入口；
      3. `发送` 动作；
      4. `开始创建` 动作。
3. FR-RC-03 结构化角色草案 `role_spec`
   1. 系统必须将用户输入收口为结构化角色草案。
   2. 当前草案至少包含：
      1. `role_name`
      2. `role_goal`
      3. `core_capabilities`
      4. `boundaries`
      5. `applicable_scenarios`
      6. `collaboration_style`
      7. `example_assets`
   3. 当关键信息不足时，系统必须明确指出缺失项，而不是直接生成创建任务。
4. FR-RC-04 启动创建门禁
   1. 当前预稿默认保留显式 `开始创建` 动作。
   2. 仅当 `role_spec` 达到最小可执行要求时，才允许触发角色创建工作流。
   3. 推断/假设（置信度：中）：
      1. 当前阶段用显式启动动作比“系统自动开始创建”更稳，避免对未收口角色描述误生成任务。
5. FR-RC-05 创建角色工作流任务生成
   1. 用户启动创建后，系统必须在任务中心生成当前角色专属的创建任务图。
   2. 创建任务图至少支持：
      1. 任务节点；
      2. 上下游依赖；
      3. 当前阶段；
      4. 任务状态；
      5. 当前选中任务详情。
   3. 当前阶段建议至少覆盖：
      1. 收集角色信息
      2. 整理角色画像
      3. 收口边界规则
      4. 初始化角色工作区
      5. 生成基础资产
      6. 校验创建完成
   4. 当前角色关联任务图的展示形态应优先继承训练中心现有“类似 `git graph` 的演进图”表达，而不是普通列表或直线卡片流。
   5. 图中任务节点状态必须同步映射任务中心中的真实运行状态，至少包含：
      1. 已完成；
      2. 运行中；
      3. 待开始；
      4. 阻塞。
   6. 页面需明确提示“当前图仅为当前角色创建工作流的投影视图，其状态真相源来自任务中心”。
6. FR-RC-06 `创建角色` 页只引用关联任务
   1. `创建角色` 页不得直接展示任务中心全量任务。
   2. 页面默认只展示当前角色创建工作流关联任务。
   3. 若用户需要全局视角，页面必须提供跳转到 `任务中心` 的入口，并自动附带当前角色/当前任务图过滤条件。
7. FR-RC-07 任务中心作为唯一任务真相源
   1. 当前角色创建工作流的任务状态、执行证据、运行结果与失败原因，必须以任务中心及其关联任务产物路径为准。
   2. `创建角色` 页面只做引用展示，不得私自维护第二套任务状态真相源。
8. FR-RC-08 新增并行 `runtime_status` 状态轴
   1. 在现有 `lifecycle_status`（`发布态 / 预发布态 / 禁训态`）之外，新增并行 `runtime_status`。
   2. `runtime_status` 固定包含：
      1. `空闲`
      2. `创建中`
      3. `训练中`
      4. `执行中`
   3. 两条状态轴必须并行展示，不得互相替换。
9. FR-RC-09 `runtime_status` 运行门禁
   1. 当角色处于 `执行中` 时，不得再被指派新普通任务。
   2. 当角色处于 `训练中` 时，不得再被指派新普通执行任务。
   3. 当角色处于 `创建中` 时：
      1. 不得被普通任务中心继续指派无关任务；
      2. 仅允许 `创建角色` 模块继续向该角色追加创建相关任务。
   4. 当角色处于 `空闲` 时，才允许恢复普通任务接单能力。
10. FR-RC-10 创建完成自动回空闲
   1. 当当前角色创建工作流关联任务全部成功完成时，系统必须自动将该角色 `runtime_status` 切回 `空闲`。
   2. 该动作不要求额外人工确认。
   3. 该动作不得自动改写该角色的 `lifecycle_status`。
11. FR-RC-11 创建失败与未完成状态处理
   1. 当任一创建任务失败或未完成时，角色必须继续保持 `创建中`。
   2. 页面必须明确展示：
      1. 当前失败/阻塞任务；
      2. 失败原因或阻塞来源；
      3. 下一步动作（重试/补充描述/跳转任务中心）。
12. FR-RC-12 创建中角色的可见性
   1. 角色一旦进入创建工作流，必须可在 `角色资产与版本` 或等效列表中被识别为“创建中”。
   2. 页面需明确区分：
      1. 当前角色是否已可用；
      2. 当前角色是否仍在创建流程中；
      3. 当前角色的版本生命周期状态。
13. FR-RC-13 图片示例展示
   1. 用户上传的图片示例必须可在当前对话区与角色草案摘要区中可见。
   2. 图片示例在当前阶段仅承担“示例参考”语义，不直接等同于最终资产。
14. FR-RC-14 创建角色页的流程展示形态
   1. `创建角色` 页必须采用与训练闭环一致的“阶段流 + 当前阶段运行态 + 当前阶段详情/任务流图”表达方式。
   2. 不得退化为纯长表单，也不得只给出普通列表式任务引用。
15. FR-RC-15 证据与留痕
   1. 创建角色工作流至少可追溯到：
      1. 用户原始输入
      2. 图片示例引用
      3. 结构化 `role_spec`
      4. 创建任务图
      5. 当前任务执行结果与证据摘要

## 5. 非功能需求（性能、安全、可用性、合规等）
1. 可理解性：
   1. 用户进入 `创建角色` 页后，应在 3 秒内理解：
      1. 当前是否仍在收集描述；
      2. 当前角色创建推进到哪一步；
      3. 当前卡在哪个任务；
      4. 下一步应该做什么。
2. 一致性：
   1. `创建角色` 页的布局、色彩、滚动方式与分栏逻辑必须继承现有 `workflow` 工作台视觉语言。
   2. 对话输入条、阶段流与任务演进图必须共处同一主工作台语义内，不得拆成多个风格割裂的子页面。
3. 可追溯性：
   1. 从 `创建角色` 页点击任务后，必须可追溯到任务中心中的同一任务对象，而不是页面本地副本。
4. 状态一致性：
   1. `runtime_status` 门禁与后端指派结果必须一致，不允许页面显示可派发、后端却拒绝。
5. 恢复能力：
   1. 页面刷新后，当前角色创建任务引用视图应能基于任务中心相关数据恢复。

## 6. 验收标准（Given/When/Then）
1. AC-RC-01 创建角色入口可见
   1. Given 用户进入 `训练中心`
   2. When 页面加载完成
   3. Then 可看到 `角色资产与版本 / 创建角色 / 训练闭环` 三个子模块
   4. And `创建角色` 可正常切换进入
2. AC-RC-02 对话与图片输入可用
   1. Given 用户进入 `训练中心 -> 创建角色`
   2. When 用户输入文本并上传图片示例
   3. Then 页面可同时展示文本轮次与图片示例
   4. And 页面可见真实可操作的输入框、图片上传入口与发送动作
   5. And 当角色已进入 `创建中` 时，输入条仍保留在当前草案页内可继续补充描述
3. AC-RC-03 结构化角色草案生成
   1. Given 用户已完成角色目标、能力、边界与示例的基础描述
   2. When 系统完成当前轮收口
   3. Then 页面可展示结构化 `role_spec`
   4. And 若仍缺关键字段，页面明确提示缺失项
4. AC-RC-04 仅生成关联任务
   1. Given 用户点击 `开始创建`
   2. When 系统生成创建任务图
   3. Then `创建角色` 页只展示当前角色创建工作流的关联任务
   4. And 不显示任务中心其他无关任务
5. AC-RC-05 git 式任务图同步状态
   1. Given 当前角色已生成创建任务图
   2. When 用户查看 `创建角色` 页中的任务图
   3. Then 任务图以接近训练中心现有 `git` 式演进图的方式展示
   4. And 节点状态与任务中心中的真实运行状态一致
6. AC-RC-06 任务真相源唯一
   1. Given 当前角色已有创建任务与执行结果
   2. When 用户在 `创建角色` 页查看某个任务
   3. Then 所见状态与任务中心中的同一任务一致
   4. And 页面不维护第二套独立任务状态
7. AC-RC-07 运行状态门禁生效
   1. Given 角色当前 `runtime_status=创建中`
   2. When 普通任务中心尝试为其指派无关任务
   3. Then 系统拒绝该指派
   4. And 仅允许 `创建角色` 模块继续追加创建相关任务
8. AC-RC-08 创建完成自动回空闲
   1. Given 当前角色创建工作流全部任务成功完成
   2. When 系统完成状态回写
   3. Then 该角色 `runtime_status` 自动变为 `空闲`
   4. And 不要求额外人工确认
9. AC-RC-09 并行状态轴可区分
   1. Given 当前角色同时存在版本生命周期状态与运行状态
   2. When 用户查看角色草案或角色详情摘要
   3. Then 页面可同时区分 `lifecycle_status` 与 `runtime_status`
   4. And 两者不会互相覆盖

## 7. 边界条件与异常处理
1. 当用户描述不足：
   1. 页面保持在收集阶段；
   2. 不得提前生成创建任务图。
2. 当图片示例上传失败：
   1. 页面需提示失败原因；
   2. 已成功上传的其他示例不得丢失。
3. 当创建任务图生成失败：
   1. 角色不得误切到 `空闲`；
   2. 页面应明确提示当前仍未进入正式创建执行阶段。
4. 当创建中任务失败：
   1. 角色继续保持 `创建中`；
   2. 页面需暴露失败任务与补救动作。
5. 当用户刷新页面或重新进入：
   1. 当前角色草案与关联任务引用视图应能恢复；
   2. 当前高亮任务与当前阶段允许回退为默认项，但不得丢失整体进度。

## 8. 依赖项与开放问题
1. 依赖项：
   1. 依赖现有 `训练中心` 继续作为角色业务域主入口。
   2. 依赖现有 `任务中心` 作为唯一任务真相源与执行底座。
   3. 依赖角色详情/角色资产列表支持新增 `runtime_status` 展示。
2. 开放问题：
   1. 当前预稿默认保留显式 `开始创建` 动作；是否最终改为“信息足够即自动生成任务”，仍待确认。
   2. 创建角色工作流的标准任务模板是否固定为 5~6 步，还是允许按角色类型动态扩展，仍待确认。
   3. 图片示例的格式、大小、张数限制当前未收口，仍待确认。
   4. 角色工作区的自动命名规则与目录落位规则当前未展开，仍待确认。
